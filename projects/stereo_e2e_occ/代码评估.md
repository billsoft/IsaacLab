# StereoOccNet 代码问题分析 & PC 训练 / Jetson Orin NX 部署隐患

---

## 第一部分：代码本身的问题

### 🔴 严重问题

#### 1. 外参定义混乱：Camera→World vs World→Camera

**位置**：`dataset.py` + `deformable_attention.py` + `occ_decoder.py`

**问题**：

代码中 `extrinsics` 定义为 **Camera→World**（即 cam2world）变换矩阵。在 `occ_decoder.py` 中：

```
R_inv = R.transpose(-1, -2)    # "逆矩阵"
t_inv = -torch.matmul(R_inv, t)
```

这是对 cam2world 矩阵求逆，得到 world2cam。然后在 `deformable_attention.py` 的 `get_reference_points()` 中用 `inv_extrinsics` 做世界→相机变换。**逻辑本身是自洽的。**

但 `position_encoding.py` 的 `FisheyeRayEncoding.get_rays_from_params()` 中直接用 `extrinsics[..., :3, :3]`（即 cam2world 的 R）把相机方向转到世界系——这是**正确的**。

**真正的风险**：`dataset.py` 中 `_camera_down_rotation()` 构建的旋转矩阵推导链较长（USD→f-theta→cam2world），且注释说"R 的结果是 f-theta +Z → World -Z"。但 f-theta 反投影给出的相机坐标系方向（Z 前、X 右、Y 下）和 `deformable_attention.py` 中实际使用的投影公式必须完全一致。如果这里有任何不匹配，**所有 3D→2D 投影都会算错，网络将无法收敛或输出垃圾**。

**建议**：写一个专门的单元测试，选几个已知 3D 点，手算它们在图像上的投影位置，和代码算出的结果对比。这是最有效的验证方式。

---

#### 2. 鱼眼投影缺少畸变系数

**位置**：`deformable_attention.py` 的 `get_reference_points()`、`position_encoding.py` 的 `get_rays_from_params()`

**问题**：

代码使用的是纯 f-theta 等距投影：`r = f × θ`。但实际的鱼眼镜头几乎总是偏离理想等距模型——Kannala-Brandt 模型用 4~5 个畸变系数（k1, k2, k3, k4）来描述真实的 `θ → r` 映射关系：

```
θ_d = θ + k1·θ³ + k2·θ⁵ + k3·θ⁷ + k4·θ⁹
r = f × θ_d
```

如果实际相机有明显的畸变系数而代码忽略了它们，投影误差在图像边缘（大角度处）会很显著。

**影响程度**：取决于数据来源。如果数据是 Isaac Sim 仿真生成的且确实使用纯等距模型，则不是问题。但如果要用在真实鱼眼相机上，这是一个**必须修复的问题**。

**建议**：在 config 中增加 `distortion_coeffs: Tuple[float, ...]` 字段，投影和反投影都加入畸变修正。

---

#### 3. `_camera_down_rotation` 使用标量 yaw 但被硬编码为 0

**位置**：`dataset.py`

**问题**：

`_build_camera_matrices()` 调用 `_camera_down_rotation(cam_yaw=0.0)`，而 `_get_extrinsics_for_frame()` 从 meta 读取 `camera_yaw_rad` 并传给同一函数。这意味着：

- 默认的 `self.intrinsics` 和 `self.extrinsics`（在 `__init__` 中构建）用的是 yaw=0
- 但 `__getitem__` 里调用 `_get_extrinsics_for_frame()` 会覆盖 extrinsics

表面看似乎可以，因为 `__getitem__` 返回的 extrinsics 是按帧计算的。但 `self.intrinsics`（在 `__init__` 中只算一次）和 `self.extrinsics`（被覆盖了所以无用）形成了一种误导性的代码结构。如果有代码路径意外使用了 `self.extrinsics` 而非按帧计算的版本，就会得到错误的结果。

**建议**：删除 `__init__` 中的 `self.extrinsics` 赋值，改为只在 `_get_extrinsics_for_frame` 中计算，避免混淆。

---

### 🟡 中等问题

#### 4. 精阶段 17,280 个查询的自注意力默认关闭

**问题说明**：

`use_fine_self_attention = False`，精阶段的 DeformableDecoderLayer 不含自注意力。这是合理的性能选择（17K 查询的自注意力非常贵），但会导致精阶段的查询之间没有信息交流，可能出现空间不连续的预测。

尾部的 `fine_spatial_conv`（Depthwise Conv3d k=3）部分弥补了这一点，但 depthwise 卷积感受野有限（仅 3×3×3）。

**建议**：如果发现精阶段预测有"棋盘格"伪影，可以考虑开启精阶段自注意力，或换用更大 kernel 的卷积（k=5 或 dilated conv）。

---

#### 5. VoxelHead 只做一次 trilinear 上采样

**问题说明**：

从 36×30×16 直接 trilinear 到 72×60×32，是 2× 上采样。这可能导致边界模糊。参考版使用多级上采样（如 Transposed Conv3d 或逐级 2×）。

**影响**：小物体可能被模糊掉，边界精度下降。

**建议**：考虑用 Conv3d + PixelShuffle3D 或 ConvTranspose3d 替代 trilinear，让网络学习上采样 kernel。

---

#### 6. Lovász-Softmax 的 O(C × N log N) 复杂度

**问题说明**：

Lovász loss 对每个类别做一次排序（`torch.sort`），体素展开后共 72×60×32 = 138,240 个元素。18 个类别 × 排序 = 性能瓶颈之一。

代码已优化为只遍历 `present_classes`（实际出现的类），但在训练数据类别丰富时仍会较慢。

**建议**：监控 Lovász loss 计算时间占比。如果超过总训练时间的 20%，可以降为每 N 步算一次，或考虑使用 Dice Loss 作为替代。

---

#### 7. DataLoader num_workers=0

**问题说明**：

默认 `num_workers=0` 是 Windows 的安全选择（Windows 的 multiprocessing 有问题），但在 Linux 上这意味着数据加载完全在主线程，GPU 会频繁等待数据。

**建议**：Linux 上设置 `num_workers=4` 或更高，加 `prefetch_factor=2`。

---

### 🟢 小问题 / 代码风格

#### 8. DNG 加载的 fallback 链

`_load_dng()` 先试 `tifffile`，再试 `rawpy`，都失败则返回全零张量。全零张量不会报错但会导致训练数据静默损坏。

**建议**：至少加 warning 日志，或在 dataset 初始化时验证至少一个库可用。

#### 9. 缺少数据增强

没有任何数据增强（随机翻转、颜色抖动、随机裁剪等）。对于小数据集，过拟合风险高。

**建议**：至少添加左右翻转（注意同步翻转体素标签和交换左右眼）。RAW 数据可以做亮度/对比度抖动。

#### 10. train.py 缺少 mIoU 评估

验证只算了 loss，没有算 mIoU（语义分割的标准指标）。loss 下降不一定代表 mIoU 提升。

**建议**：在 validate 中增加 confusion matrix 统计，计算 per-class IoU 和 mIoU。

## 第二部分：PC 训练 → Jetson Orin NX 16GB 部署隐患

### 🔴 严重隐患

#### 1. 模型大小 vs Orin NX 16GB 显存

**分析**：

根据 `verify_shapes.py` 的实测，模型参数量约 **~2.07M**，FP32 约 8MB。这不是问题。

但推理时的**激活值和中间张量**才是瓶颈：

| 张量 | 尺寸 | FP16 显存 |
|------|------|----------|
| 输入图像 | 2×1×1080×1280 | ~5.3 MB |
| PatchEmbed 输出 | 2×128×68×80 | ~2.8 MB |
| Encoder 中间（窗口注意力 QKV） | ~3×5440×128×2相机 | ~8 MB |
| 粗阶段 grid_sample | 2×8×960×4 采样点 | 小 |
| 精阶段 grid_sample | 2×8×17280×4 采样点 | **~4.4 MB** |
| fine query 中间 | 17280×128 | ~4.4 MB |
| VoxelHead Conv3d | 72×60×32×各通道 | ~数 MB |

**总推理显存估计**：FP16 下约 **1.5~3 GB**。Orin NX 16GB 够用，但需要注意 Orin NX 的 GPU 和 CPU 共享内存——系统 + 其他进程也会占用。实际可用 GPU 显存可能只有 10~12 GB。

**建议**：在 Orin NX 上实际跑一次 benchmark，监控 `tegrastats` 的显存使用。

---

#### 2. grid_sample 在 TensorRT 上的兼容性

**这是最大的部署风险。**

`F.grid_sample` 在 PyTorch 中使用方便，但在 TensorRT 部署时是著名的问题算子：

- TensorRT 对 `grid_sample` 的支持不完整（特别是 `align_corners=False` + `padding_mode='zeros'` 的组合）
- 某些 TensorRT 版本需要用自定义 plugin 实现
- ONNX export 时 `grid_sample` 的 opset 版本要求（opset 16+）

本网络中 `grid_sample` 出现在**三个地方**：
1. `deformable_attention.py`：可变形交叉注意力采样（核心路径）
2. `temporal_fusion.py`：ego-motion 对齐的 3D grid_sample
3. `voxel_head.py`：trilinear 插值（但这个可以用 TRT 原生 resize 替代）

**建议**：
- 确认目标 JetPack 版本的 TensorRT 是否支持所用的 grid_sample 变体
- 准备 TensorRT plugin 作为后备方案
- 考虑在导出前将 grid_sample 替换为手写的双线性插值（用 `gather` 实现）

---

#### 3. Orin NX 不支持 BF16，FP16 精度问题

Orin NX（SM 8.7 Ampere 架构）支持 FP16 和 INT8，但不支持 BF16。

代码中 Lovász loss 已经做了 FP32 fallback，但推理时如果全程 FP16：

- `torch.atan2` 在 FP16 下精度有限，可能影响鱼眼投影精度
- 大角度（接近 FOV 边缘）的 `sin`/`cos` 精度下降
- `torch.linalg.inv`（时序融合中）在 FP16 下可能数值不稳定

**建议**：投影相关的计算（参考点计算、射线编码）保持 FP32 或在 TensorRT 中标记为 FP32 层。

---

#### 4. 算子兼容性：`torch.linalg.inv` 和 `scaled_dot_product_attention`

- `torch.linalg.inv`：TensorRT 不支持，需要替换为解析求逆（代码中 occ_decoder 已经这样做了，但 temporal_fusion 中的 ego_motion 求逆仍用 `torch.linalg.inv`）
- `F.scaled_dot_product_attention`：较新的 PyTorch 特性，ONNX export 和 TensorRT 支持需要确认
- `torch.meshgrid(indexing='ij')`：ONNX 导出时可能有问题

**建议**：在部署前做一次完整的 ONNX export 测试，检查所有不支持的算子。

---

### 🟡 中等隐患

#### 5. DNG/RAW 解码的 CPU 开销

Orin NX 的 CPU（6 核 Cortex-A78AE）性能远不如 PC。`tifffile` 或 `rawpy` 解码 12-bit DNG（1080×1280）在 ARM CPU 上会很慢。

**建议**：
- 预处理阶段将 DNG 转为 numpy `.npy` 或直接用 MIPI CSI 接口获取 RAW 数据
- 如果用实际相机，考虑用 ISP 硬件管线直接输出 RAW buffer，绕过 DNG 文件
- 用 Orin NX 的 NVJPG/NVDEC 硬件加速（但 DNG 不在支持列表内，需要自行处理）

---

#### 6. Conv3d 在 TensorRT 上的效率

3D 卷积在 GPU 上的效率通常不如 2D 卷积。在 Orin NX 这种移动级 GPU 上，VoxelHead 的多个 Conv3d 可能成为瓶颈。

**涉及的 Conv3d 层**：
- `fine_spatial_conv`：Depthwise Conv3d(128, 128, k=3, groups=128) — 这是 depthwise，TensorRT 可能不会很好地优化分组 3D 卷积
- `reduce`：Conv3d(128→64→32, k=3)
- `refine`：Conv3d(32→32, k=3)
- `cls_head`：Conv3d(32→18, k=3)

**建议**：用 `trtexec` profiler 逐层分析延迟。如果 Conv3d 占比过高，可以考虑将 3D 卷积拆解为 2D 卷积 + 1D 卷积的伪 3D 结构。

---

#### 7. 窗口注意力的 Padding 操作

`WindowAttention` 中有动态 padding（`F.pad`），这会导致 ONNX 图中出现动态 shape，TensorRT 需要设置 dynamic shape profile。

68 % 7 = 5（需要 pad 2），80 % 7 = 3（需要 pad 4）。这些是固定的，但 ONNX 导出时可能不会被常量折叠。

**建议**：导出前将 padding 硬编码为固定值。

---

#### 8. PC 训练 vs Orin 推理的数值一致性

在 PC 上用 FP32/AMP 训练的模型，部署到 Orin NX 上用 FP16/INT8 推理时，可能出现精度下降：

- INT8 量化需要标定（calibration dataset），没有做的话精度可能大幅下降
- FP16 推理中 softmax 的数值范围和 PC 上的 FP32 不同
- BatchNorm 的 running_mean/running_var 在 FP16 下可能溢出

**建议**：先跑 FP16 推理对比 FP32 的 mIoU 差异。如果差异 > 1%，需要量化感知训练（QAT）或混合精度部署（关键层保持 FP32）。

---

### 🟢 一般隐患

#### 9. 推理延迟预期

Orin NX 16GB 的 GPU 算力约 **100 TOPS (INT8) / 21 TFLOPS (FP16)**，远低于 PC 端的 RTX 3090（142 TFLOPS FP16）或 4090（165 TFLOPS FP16）。

粗略估计：如果 PC 上 FP16 推理 50ms，Orin NX 上可能需要 300~500ms（FP16）。对于机器狗巡检等非实时场景可能够用，但如果需要 10+ FPS 的实时性，大概率不够。

**建议**：明确目标帧率。如果需要实时，考虑减小 fine_size（如 18×15×8）或减小 embed_dim（如 64）。

---

#### 10. JetPack / CUDA / cuDNN 版本匹配

Orin NX 使用 JetPack SDK（目前最新 6.x 系列），内含特定版本的 CUDA、cuDNN、TensorRT。PC 训练环境的 PyTorch 版本和 CUDA 版本必须与 JetPack 兼容。

**常见坑**：
- PC 上用 CUDA 12.x 训练，Orin NX 的 JetPack 可能只支持 CUDA 12.2
- PyTorch for Jetson 需要从 NVIDIA 官方下载 ARM wheel，不能直接 pip install
- ONNX opset 版本和 TensorRT parser 的兼容性

**建议**：在项目早期就确认 JetPack 版本，并在 PC 上安装对应版本的 CUDA/cuDNN。

---

## 总结：优先级排序

### 必须修复（否则可能不能正确训练或部署）

1. **验证外参/投影正确性**：写单元测试，已知 3D 点 → 图像投影 → 对比手算结果
2. **grid_sample TensorRT 兼容性**：确认目标 JetPack 版本支持情况
3. **temporal_fusion 中的 torch.linalg.inv**：替换为解析求逆
4. **完整 ONNX export 测试**：在 PC 上导出并用 `onnxruntime` 验证

### 强烈建议（提升训练效果和部署效率）

5. 添加数据增强
6. 添加 mIoU 评估指标
7. 鱼眼畸变系数支持（如果目标是真实相机）
8. FP16 推理精度验证
9. Orin NX 实际 benchmark

### 可以后续优化

10. VoxelHead 用学习式上采样替代 trilinear
11. 精阶段空间一致性卷积增大感受野
12. Linux 上增加 DataLoader workers
13. Conv3d 效率优化（伪 3D 拆解）
