# StereoOccNet 代码问题分析 & PC 训练 / Jetson Orin NX 部署隐患

---

## 第一部分：代码本身的问题

### 🔴 严重问题

#### 1. 可变形注意力中相机维度的 for 循环是性能瓶颈

**文件：** `deformable_attention.py` 第 121-152 行

```
for cam in range(N):  # N=2, 每次循环一次 grid_sample
```

虽然已经把 head 维度合并到 batch 做了优化，但**仍然对 2 个相机做串行 for 循环**。每次循环内的 `grid_sample` 是一个独立的 CUDA kernel launch，而且循环体内有大量的 view/permute/reshape 操作。

**影响：** 在 Jetson Orin NX 上，CUDA kernel launch 的 overhead 比桌面 GPU 大得多。2 个相机 × 4 层 decoder（2 粗+2 精）= 8 次串行 grid_sample 调用。

**建议修复：** 将两个相机的 value 和 locs 在 batch 维度拼接，做**一次** grid_sample：

```python
# 把 N 个相机合并到 batch: [B*N*num_heads, head_dim, H, W]
# 一次 grid_sample，然后 reshape 回来按相机分拆、加权求和
```

这能把 grid_sample 调用从 N 次减少到 1 次。

#### 2. 外参逆矩阵使用 `torch.inverse()` — 数值不稳定且低效

**文件：** `occ_decoder.py` 第 108 行

```python
inv_extrinsics = torch.inverse(extrinsics) if extrinsics is not None else None
```

**问题：**
- `torch.inverse()` 对于 4×4 矩阵数值稳定性不如 `torch.linalg.inv()`
- 更重要的是，外参矩阵是刚体变换（旋转+平移），其逆矩阵有**解析形式**，不需要做通用矩阵求逆：

```python
# R^T 就是旋转的逆，平移的逆 = -R^T @ t
R = extrinsics[..., :3, :3]
t = extrinsics[..., :3, 3:]
R_inv = R.transpose(-1, -2)
t_inv = -R_inv @ t
# 组装逆矩阵
```

**影响：** 在 FP16 推理时，通用矩阵求逆更容易出现数值问题。

#### 3. `deformable_attention.py` 中参考点投影缺少 FOV 有效性掩码

**文件：** `deformable_attention.py` 第 50-87 行 `get_reference_points()`

f-theta 投影计算了 θ 和像素坐标，但**没有检查**：
- θ 是否超出 FOV 范围（max_fov_deg=157.2° → θ_max ≈ 1.372 rad）
- 投影点是否在相机后方（cam_z < 0）
- 归一化坐标是否超出 [-1, 1] 范围

虽然 `grid_sample(padding_mode='zeros')` 对超出范围的采样返回零，但**这些无效的参考点仍会产生注意力权重**。网络需要自己学习忽略这些无效点，增加了学习难度。

**建议：** 对超出 FOV 的参考点生成一个布尔掩码，将对应的 attention_weights 强制置零。

#### 4. 训练脚本未对 `scaler` 状态做 checkpoint 保存/恢复

**文件：** `train.py`

保存了 model、optimizer、epoch，但**没有保存** `scaler.state_dict()`。

**影响：** 使用 `--amp --resume` 恢复训练时，GradScaler 的动态 loss scale 会重置为初始值（65536），可能导致恢复后几个 step 出现 NaN/Inf，loss scale 要重新收敛。

**修复：** 保存时加 `'scaler': scaler.state_dict()`，恢复时加 `scaler.load_state_dict(ckpt['scaler'])`。

#### 5. 训练脚本未保存 `scheduler` 状态

**文件：** `train.py`

`CosineAnnealingLR` 的状态没有 checkpoint。恢复训练后，学习率调度器会从第 0 个 epoch 重新开始余弦衰减，而不是从断点 epoch 继续。

**影响：** 如果在 epoch 50 断点，恢复后学习率应该在余弦曲线的中段，但实际会跳回最高点。这会严重破坏训练轨迹。

### 🟡 中等问题

#### 6. 数据集中 `_camera_down_rotation` 是 @staticmethod 但返回 Tensor 而非 NumPy

这个方法在 `_build_camera_matrices` 和 `_get_extrinsics_for_frame` 中都被调用，它返回 `torch.Tensor`，但输入的 `cam_yaw` 是 float，内部用了 `np.cos/np.sin`。混用 NumPy 和 PyTorch 不会出错，但不够干净。

#### 7. `_get_extrinsics_for_frame` 中缺少 X/Y 偏移

**文件：** `dataset.py` 第 157-177 行

代码读取了 `cam_pos = meta.get('camera_pos', [0.0, 0.0, 3.0])`，但只使用了 `cam_pos[2]`（高度），**忽略了 X 和 Y 分量**。

如果数据采集时相机有水平移动（Isaac Sim 中随机位移），体素局部坐标原点应该在相机正下方。如果没有正确设置 X/Y 平移，体素内容和标签会错位。

**需要确认：** 体素标签是否已经以相机投影点为原点？如果是，则不需要 X/Y 偏移；如果不是，这就是一个**严重的标签对齐 bug**。

#### 8. Lovász-Softmax 对全部体素做 softmax 然后 flatten — 显存高峰

**文件：** `loss.py` 第 46 行

```python
prob = F.softmax(pred, dim=1)
prob_flat = prob.permute(1, 0, 2, 3, 4).reshape(C, -1)
```

这里对完整的 `[B, 18, 72, 60, 32]` 张量做 softmax 并 flatten，产生一个 `[18, B×72×60×32]` = `[18, 138240]` 的张量。同时 `torch.sort` 在后面对每个类的 138K 个元素排序。

**影响：** 显存峰值大约增加 20MB（不算大），但在 Orin NX 上每一点都要省。

**建议：** 可以用 amp 排除 Lovász Loss 的 softmax 运算（保持 FP32），或者对 Lovász 做体素子采样。

#### 9. `image_encoder.py` 对相机逐个处理

```python
for i in range(N):
    x_cam = x[:, i:i + 1]
    ...
```

两个相机是独立的 Encoder，共享参数但串行执行。可以把两个相机合并到 batch 维度并行处理：`[B*N, H, W, C]`，减少 Python 层面的循环。

#### 10. `temporal_fusion.py` 中 `align_memory` 的 grid_sample 使用 `align_corners=True`

而 `deformable_attention.py` 和 `voxel_head.py` 中的 `grid_sample` 和 `interpolate` 使用 `align_corners=False`。

**影响：** 两种对齐模式对边界体素的处理方式不同，可能导致时序 warp 后的体素位置有约半个体素的偏移。在 10cm 分辨率下就是 5cm 的错位。

### 🟢 小问题

#### 11. `prepare_splits.py` 的 shuffle 破坏了时序连续性

随机打乱后再切分，训练集和验证集的帧可能是交替的。如果开启时序模式，连续帧分属不同 split 会导致数据泄露。

**建议：** 按时间块切分，而不是逐帧 shuffle。

#### 12. `raw_embed.py` 中最后一个卷积是 stride=1

```python
nn.Conv2d(128, embed_dim, kernel_size=3, stride=1, padding=1),
```

如果 `embed_dim != 128`，这一层改变通道数但不降采样。当前 config 中 embed_dim=128，所以这层实际上是 128→128 的恒等维度变换，有少许冗余参数（约 147K）。

#### 13. 没有数据增强

训练脚本中完全没有数据增强（随机翻转、颜色抖动、RAW 噪声注入等）。对于仿真数据集，sim-to-real gap 本身就大，缺少增强会导致过拟合。

## 第二部分：PC 训练 → Jetson Orin NX 部署隐患

### 🔴 严重隐患

#### A. grid_sample 在 Jetson 上 TensorRT 转换的兼容性问题

`F.grid_sample` 在 TensorRT 中的支持历来有坑：
- TensorRT 7.x 不支持 5D grid_sample（时序模块中用到）
- TensorRT 8.x 支持 4D 但 `align_corners` 参数行为可能与 PyTorch 不一致
- Jetson Orin NX 上常用的 JetPack 6.x 内置 TensorRT 8.6+，基本 4D 可用但需要测试

**temporal_fusion.py 的 3D grid_sample（5D 输入）在 TensorRT 中很可能需要手写插件或放在 PyTorch 端执行。**

**建议：** 部署时如果用 TensorRT，考虑把时序模块和主模型分开处理。或者直接用 torch2trt / torch_tensorrt 并验证每个 op 的支持情况。

#### B. `torch.linalg.inv` 和 `torch.inverse` 在 TensorRT 中不支持

Decoder 中的外参求逆操作在 TensorRT 中没有对应算子。

**建议：** 预计算外参逆矩阵，作为输入传入模型，而不是在模型内部动态计算。或者改用解析公式。

#### C. FP16 推理时 f-theta 投影的数值精度

`get_reference_points()` 中的 `arctan2`、`sqrt`、`cos`、`sin` 链式运算在 FP16 下精度损失可能导致投影坐标偏移。

**量化估算：** FP16 的有效精度约 3 位十进制小数。对于 1280 像素宽度的归一化坐标，3 位精度意味着约 1.28 像素的量化误差。这在 stride=16 的特征图上相当于 ~0.08 个特征像素的偏移，影响不大，但在边缘区域（θ 接近 FOV 极限）会更严重。

**建议：** 参考点计算保持 FP32。

#### D. Orin NX 16GB 的显存限制

**推理显存估算（batch_size=1）：**

| 组件 | 大约显存 |
|------|---------|
| 模型参数 (FP16) | ~6MB |
| 输入图像 2×1×1080×1280 (FP16) | ~5.3MB |
| 特征图 2×128×68×80 (FP16) | ~2.8MB |
| Decoder 中间态 (粗+精查询) | ~50MB |
| grid_sample 临时缓存 | ~30MB |
| VoxelHead 中间态 | ~40MB |
| **推理总计** | **~150-200MB** |

**16GB 对于推理完全够用**，即使加上系统和其他进程的开销也毫无压力。

**训练显存估算（batch_size=1）：**

| 组件 | 大约显存 |
|------|---------|
| 推理占用 | ~200MB |
| 梯度 (≈参数量) | ~6MB |
| 优化器状态 (Adam: 2×参数) | ~12MB |
| 激活值缓存 (反向传播用) | ~2-4GB |
| **训练总计** | **~3-5GB** |

在 Orin NX 上训练也是可行的（开 AMP + gradient checkpoint），但**不建议**——Orin 的 GPU 算力（Ampere 架构，带宽有限）训练效率远低于桌面 GPU。

### 🟡 中等隐患

#### E. PyTorch 版本一致性

PC 训练通常用 CUDA 12.x + PyTorch 2.x。Jetson Orin NX 使用 JetPack 提供的 PyTorch wheel，版本可能滞后（例如 JetPack 6.0 配 PyTorch 2.3）。

**潜在问题：**
- `torch.amp.autocast('cuda')` 的 API 在 PyTorch 2.0 前后有变化
- `F.scaled_dot_product_attention` 需要 PyTorch 2.0+
- `torch.linalg.inv` 需要 PyTorch 1.9+

**建议：** 在 PC 和 Jetson 上使用**完全相同的 PyTorch 版本**。或者导出 ONNX 后用 TensorRT 部署。

#### F. DNG 文件加载依赖 tifffile/rawpy

Jetson 上这两个库可能需要从源码编译。rawpy 依赖 LibRaw C 库，在 ARM Ubuntu 上编译可能需要额外配置。

**建议：** 如果是实际部署（而非从文件推理），应该直接从相机驱动获取 RAW 数据，绕过 DNG 文件格式。

#### G. 模型中动态 shape 操作

以下操作在 ONNX/TensorRT 导出时可能导致问题：
- `config.feat_size` 用 `math.ceil` 动态计算
- `FisheyeRayEncoding.get_rays_from_params()` 中的 `torch.meshgrid` 配合动态 H, W
- `WindowAttention` 中的 pad 操作

**建议：** 导出 ONNX 时固定所有 shape 为静态值（batch_size=1, H=1080, W=1280）。

### 🟢 小隐患

#### H. `weights_only=False` 的安全风险

`torch.load(..., weights_only=False)` 允许反序列化任意 Python 对象。在部署环境中加载不可信的 checkpoint 文件有安全风险。

**建议：** 部署时只加载 `state_dict`，config 用代码写死或从 JSON 加载。

#### I. 推理脚本的 benchmark 模式使用单位矩阵做外参

`inference.py` 的 benchmark 中外参为单位矩阵，但实际推理的外参包含旋转。单位矩阵外参求逆后仍是单位矩阵，不会暴露数值问题。**benchmark 得到的延迟数值是可靠的**（因为 shape 相同，计算量相同），但不能验证数值正确性。

## 第三部分：综合结论

### 代码质量评估

整体代码质量**良好**，架构清晰，模块化程度高，注释充分。核心算法实现正确，管线对齐验证脚本（verify_shapes.py）非常专业。

**需要立即修复的问题：**
1. 训练 checkpoint 缺少 scaler 和 scheduler 状态（**问题 4、5**）— 影响断点续训
2. 确认 `_get_extrinsics_for_frame` 的 X/Y 偏移是否正确（**问题 7**）— 可能导致标签错位
3. 外参求逆改用解析公式（**问题 2**）— 数值稳定性 + 部署兼容性

**建议后续优化：**
1. 合并相机维度的 grid_sample（**问题 1**）— 推理加速
2. 添加 FOV 有效性掩码（**问题 3**）— 提升收敛速度
3. 统一 align_corners 设置（**问题 10**）

### PC 训练 → Jetson 部署路径建议

```
PC 训练 (CUDA GPU)
    ↓ 保存 best_model.pth
    ↓
PC 上导出 ONNX (固定 shape, batch=1)
    ↓ 预计算外参逆矩阵，从模型中移除 torch.inverse
    ↓
Jetson 上 trtexec 转换 TensorRT engine
    ↓ 逐层验证 FP16 精度
    ↓
Jetson 部署 (C++ 或 Python TensorRT runtime)
```

如果不走 TensorRT 路线，也可以直接用 PyTorch + torch.jit.trace 在 Jetson 上跑 FP16 推理，延迟会略高但兼容性更好。

**底线判断：这个模型在 Jetson Orin NX 16GB 上推理没有显存瓶颈，关键挑战在于 TensorRT 算子兼容性和 FP16 数值精度。**
