# StereoOccNet 数据集升级规范：支持运动预测与轨迹预测

> **背景**：当前数据集只支持单帧语义体素预测。本文档定义完整的数据集升级方案，使其支持**体素运动方向/速度预测**和**物体级未来轨迹预测**。所有新增数据都可以从 Isaac Sim 仿真中直接获取。

---

## 一、为什么需要实例分组（Instance ID）

### 1.1 三个层级的预测能力

| 层级 | 输出 | 需要的数据 | 能力 |
|------|------|----------|------|
| ① 语义分类 | 每个体素的类别（18 类） | 语义体素（已有） | "这是什么" |
| ② 逐体素 Flow | 每个体素的速度向量 (vx, vy) | 语义 + **flow 真值** | "往哪走" |
| ③ 物体轨迹预测 | 每个物体未来 N 帧的位置序列 | 语义 + flow + **实例 ID** + **物体属性** | "未来 3 秒在哪里" |

**没有实例 ID 时**：你只能做逐体素 flow——知道"这个体素在移动"，但不知道"这个体素属于哪辆车"。如果一辆车占了 200 个体素，你无法把这 200 个 flow 向量关联起来做一致性约束。

**有了实例 ID 后**：

- 同一辆车的 200 个体素共享同一个 instance_id，它们的 flow 应该高度一致——可以加**实例级 flow 一致性损失**
- 可以计算**每个物体的质心轨迹**，做物体级轨迹预测
- 可以区分"两辆紧邻的车各自往哪走"——纯语义（都是"车"类）无法区分
- 未来做 planning（路径规划）时，需要知道"第 3 辆车会不会挡住我的路"——必须有实例级理解

### 1.2 Isaac Sim 中获取实例 ID 的成本

**几乎为零。** Isaac Sim 中每个 NPC/Actor 已经有唯一的 prim path。你在生成语义体素时已经遍历了每个 prim 的几何体来判断体素占用——只需在同一个循环里额外记录"这个体素被哪个 prim 占据"。

## 三、各文件详细规范

### 3.1 语义体素（已有，不变）

**文件**：`voxel/frame_XXXXXX_semantic.npz`

| 字段 | 类型 | 形状 | 值域 | 说明 |
|------|------|------|------|------|
| `data` | uint8 | (72, 60, 32) | 0~17, 255 | 语义类别 ID，255=未观测 |

### 3.2 🆕 实例 ID 体素

**文件**：`voxel/frame_XXXXXX_instance.npz`

| 字段 | 类型 | 形状 | 值域 | 说明 |
|------|------|------|------|------|
| `data` | uint16 | (72, 60, 32) | 0~65534, 65535 | 实例 ID，0=背景/静态，65535=未观测 |

**设计决策**：

| 决策 | 理由 |
|------|------|
| uint16 而非 uint8 | 单帧最多 255 个实例太少（密集场景可能有几百个物体） |
| 0 = 背景/静态 | 地面、建筑、植被等不需要逐实例区分 |
| 65535 = 未观测 | 和语义的 ignore_index=255 对应 |
| 每帧独立编号 | 帧间实例对应关系由 `trajectory/*.json` 处理 |

**Isaac Sim 中的生成方式**：

```
对每个体素 (x, y, z):
    if 被某个动态 prim 占据:
        instance_grid[x,y,z] = prim_to_instance_id[prim_path]
    elif 被静态物体占据:
        instance_grid[x,y,z] = 0  # 背景
    else:
        instance_grid[x,y,z] = 65535  # 未观测/空气
```

**实例 ID 分配规则**：

| ID 范围 | 含义 | 示例 |
|---------|------|------|
| 0 | 背景（所有静态物体合并） | 地面、建筑、围栏、植被 |
| 1~999 | 车辆实例 | 轿车 #1, SUV #2, ... |
| 1000~1999 | 行人实例 | 行人 #1000, ... |
| 2000~2999 | 两轮车实例 | 自行车 #2000, ... |
| 3000~9999 | 其他动态物体 | 施工设备 #3000, ... |
| 65535 | 未观测 | 看不到的区域 |

> 分段编号不是必须的，但方便调试——看到 ID=1005 就知道是第 6 个行人。

### 3.3 🆕 运动向量（Flow）体素

**文件**：`voxel/frame_XXXXXX_flow.npz`

| 字段 | 类型 | 形状 | 单位 | 说明 |
|------|------|------|------|------|
| `flow` | float16 | (72, 60, 32, 2) | m/s | 体素局部坐标系下的 (vx, vy) |
| `flow_mask` | uint8 | (72, 60, 32) | 0/1 | 1=有效 flow（动态物体），0=无效（静态/空气） |

**关键字段说明**：

**flow (vx, vy)**：体素在**当前帧体素局部坐标系**下的瞬时速度。

| 方向 | 正值含义 | 负值含义 |
|------|---------|---------|
| vx | 世界 +X 方向（前） | 世界 -X 方向（后） |
| vy | 世界 +Y 方向（左） | 世界 -Y 方向（右） |

> 为什么只存 2D (vx, vy) 不存 vz？ 在俯视场景中，物体主要在地面平面运动，Z 方向速度接近零。存 2D 节省 33% 存储。如果将来有飞行器等 Z 方向运动的场景，改成 (72,60,32,3) 即可。

**flow_mask**：区分"速度真的是 0（静止物体）"和"这个体素没有 flow 信息（空气/未观测）"。

| 场景 | flow 值 | flow_mask | 训练时处理 |
|------|---------|-----------|----------|
| 静止的车 | (0.0, 0.0) | **1** | 参与 loss（预测应为 0） |
| 行驶的车 | (1.5, -0.3) | **1** | 参与 loss |
| 地面/建筑 | (0.0, 0.0) | **0** | **不参与 loss** |
| 空气/未观测 | (0.0, 0.0) | **0** | **不参与 loss** |

> 这个区分非常重要。如果把地面体素也算进 flow loss，网络会被海量的零值主导，学到"所有东西都不动"的偏见。flow_mask=1 只包含动态物体（无论它此刻是否在动），让网络专注于学习真正重要的运动预测。

**Isaac Sim 中的生成方式**：

```
对每个动态 prim (NPC/Actor):
    velocity = prim.get_linear_velocity()    # (vx, vy, vz) 世界坐标系 m/s
    occupied_voxels = 该 prim 在体素网格中占据的所有格子
    flow_grid[occupied_voxels, 0] = velocity[0]   # vx
    flow_grid[occupied_voxels, 1] = velocity[1]   # vy
    flow_mask[occupied_voxels] = 1

静态 prim 不处理（flow 默认 0.0，mask 默认 0）
```

**注意坐标系**：Isaac Sim 的世界坐标系和体素局部坐标系可能有偏移（体素原点 = 相机地面投影点）。但速度是**矢量**，和原点位置无关——只要两个坐标系的轴方向一致，速度可以直接用。如果相机有 yaw 旋转，需要把速度旋转到体素局部坐标系：

$$v_{local} = R_z(-\text{yaw}) \cdot v_{world}$$

| 时间范围 | 采样间隔 | 点数 | 用途 |

## 五、文件大小估算

以单帧为例，体素网格 72×60×32 = 138,240 个体素：

| 文件 | 原始大小 | 压缩后（npz） | 说明 |
|------|---------|-------------|------|
| semantic.npz | 138,240 × 1B = 135KB | ~20KB | uint8，稀疏（大量空气=0） |
| **instance.npz** | 138,240 × 2B = 270KB | ~30KB | uint16，大部分是 0（背景） |
| **flow.npz** (flow+mask) | 138,240 × (4+1)B = 660KB | ~50KB | float16×2 + uint8 mask，动态体素稀疏 |
| **traj.json** | — | ~2-5KB | 取决于物体数量 |
| meta.json (升级) | — | ~0.5KB | 文本，很小 |
| **单帧新增总计** | — | **~85KB** | 相比 DNG (2.6MB×2)，增量很小 |

1000 帧的数据集新增存储：~85MB。相比原有 DNG 数据（~5.2GB）只增加 1.6%。

## 六、数据验证脚本规范

升级后建议编写验证脚本检查数据完整性：

### 6.1 必须验证的项目

| 检查项 | 规则 | 严重性 |
|--------|------|--------|
| semantic 和 instance 形状一致 | 都是 (72,60,32) | 致命 |
| flow 形状正确 | (72,60,32,2) | 致命 |
| flow_mask 形状正确 | (72,60,32) | 致命 |
| instance_id 在注册表中 | 除 0 和 65535 外 | 严重 |
| 动态体素有 flow | `semantic ∈ 动态类 AND instance > 0` → `flow_mask=1` | 严重 |
| 静态体素无 flow | `semantic ∈ 静态类` → `flow_mask=0` | 严重 |
| 同实例 flow 一致性 | 同一 instance_id 的所有体素 flow 应接近 | 警告 |
| timestamp 单调递增 | `t[i+1] > t[i]` | 致命 |
| 帧间时间差合理 | `dt ≈ sim_step_sec ± 10%` | 警告 |
| DNG + semantic + instance + flow 齐全 | 每帧四件套 | 严重 |
| traj 中的 instance_id 在 instance.npz 中存在 | 交叉验证 | 严重 |
| future_positions 时间点正确 | `t > 0` 且递增 | 严重 |
| bbox_size 合理 | 车 > 1m，行人 > 0.3m | 警告 |
| flow 值范围 | 绝对值 < 30 m/s（极端情况） | 警告 |

## 七、和业界标准数据集的对比

| 维度 | nuScenes | Waymo Open | **Ours (v2)** |
|------|----------|------------|--------------|
| 传感器 | 6 相机 + LiDAR | 5 相机 + LiDAR | 2 鱼眼相机（RAW） |
| 图像格式 | JPEG | JPEG | **DNG (12-bit RAW)** |
| 体素分辨率 | 0.4m | 0.1m | **0.1m** |
| 语义类别 | 16 | 23 | 18 |
| **实例 ID** | ✅ | ✅ | ✅ (新增) |
| **Flow 真值** | ✅ (LiDAR-based) | ✅ (LiDAR-based) | ✅ (仿真直接获取) |
| **轨迹真值** | ✅ (检测+跟踪) | ✅ (人工标注) | ✅ (仿真直接获取) |
| 未来轨迹时长 | 6s | 8s | **3s** (可配置) |
| 遮挡标注 | 近似 | ✅ | ✅ (仿真精确计算) |

> 你的数据集虽然是仿真的，但**精度反而更高**——nuScenes 和 Waymo 的 flow 真值是从 LiDAR 推算的（有噪声），你的是从仿真器直接读取的物理量（精确值）。

## 八、Isaac Sim 采集脚本升级要点

### 8.1 在现有采集循环中新增的数据获取

```
已有步骤（保持不变）:
    1. 渲染左右眼 DNG
    2. 获取体素语义标签
    3. 保存相机位姿

新增步骤:
    4. 遍历所有动态 prim:
        a. 读取 instance_id（从 prim_path 映射）
        b. 读取 linear_velocity, angular_velocity
        c. 读取 bbox (world_transform → center + size)
        d. 填充 instance_grid 和 flow_grid
    5. 读取当前帧所有 prim 的 world_position → 存入 traj
    6. 从仿真器 timeline 读取 future_positions（回放/预计算）
    7. 保存 instance.npz, flow.npz, traj.json
```

### 8.2 future_positions 的获取方式

| 方式 | 适用场景 | 精度 |
|------|---------|------|
| **后处理**：先完整跑一遍仿真录制所有帧的位置，再回填 future | 最简单 | 100% 准确 |
| 实时预计算：在帧 t 时让仿真器快进 N 步读位置再回退 | 复杂但一趟完成 | 100% 准确 |
| 匀速外推：用当前 velocity × Δt | 最简单但不准 | 直线运动准，转弯差 |

**推荐后处理方式**——先把所有帧的位姿都存下来，再写一个脚本从全局时间线中提取每帧的 past/future positions。

## 九、向后兼容性

### 9.1 旧版数据仍可用

| 组件 | v1 数据（只有 semantic） | v2 数据（完整） |
|------|----------------------|---------------|
| 语义训练 | ✅ 正常工作 | ✅ |
| 时序训练 | ✅（如果帧连续） | ✅ |
| Flow 训练 | ❌（缺 flow.npz） | ✅ |
| 轨迹训练 | ❌ | ✅ |

### 9.2 Dataset 代码的 fallback 策略

```
def _load_flow(self, frame_id):
    path = .../flow/frame_XXXXXX_flow.npz
    if exists(path):
        return load(path)     # 有 flow 数据
    else:
        return zeros(72,60,32,2), zeros_mask(72,60,32)  # 没有 flow → 全零 + mask=0
```

> flow_mask 全零意味着所有体素不参与 flow loss——自动降级为纯语义训练。不需要手动区分 v1/v2 数据。
